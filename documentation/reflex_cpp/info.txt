	@WString Description: "#p Reflex serialization provides a lightweight, explicit way to encode and decode strongly-typed data to and from a binary stream.
#p It is designed for deterministic layouts, low overhead, and full control over what is written.

#p Unlike PropertySet serialization (which targets structured, schema-light data), this API is intended for **compact, ordered, binary serialization** of known data layouts.

#h1 Archive

#p Data::Archive is a byte container used as the backing store for serialization.
#p It is a simple typedef over Array<UInt8>, and can be treated as a writable or readable byte stream.

<code>
#p Data::Archive outstream;
</code>

#p To read from an archive, use Data::Archive::View, which maintains a read cursor without copying data:

<code>
#p Data::Archive::View instream = outstream;
</code>

#p Archive::View performs deserialization directly from the underlying byte buffer. For simple value types, data is read by copying from the current stream position and advancing an internal read cursor. This avoids additional heap allocations and minimizes intermediate copying, making the operation suitable for performance-sensitive code paths.


#h1 Writing data (Serialize)

#p Use Data::Serialize to append strongly-typed values to an archive in sequence.

<code>
#p Data::Serialize(stream, 1, 2.0f);
#p Data::SerializeUTF8(stream, L$qwide$q);
</code>

#p Values are written in the order provided. The resulting stream is compact and contains no metadata beyond what is required to decode variable-length data (such as strings).


#h1 Reading data (Deserialize)

#p Use Data::Deserialize to read values back in the same order they were written.

<code>
#p auto [i, f] = Data::Deserialize<Int32, Float32>(stream);
#p auto wstring = Data::DeserializeUTF8(stream);
</code>

#p Deserialization advances the read cursor stored in the Archive::View.

#p The template parameters define the expected types and enforce strong typing at the call site.

#h2 Validation

#p Within Reflex, deserialization fundamentally expects pre-validated input. Individual Data::Deserialize calls do not perform size or integrity checks - this is by design, not omission. A size check at the individual deserializer level is largely meaningless in isolation: confirming that 4 bytes remain before reading a UInt32 says nothing about whether that value is semantically correct. 
#p The general approach therefore is validation at the stream boundary, before deserialization begins. Once a valid, correctly bounded Archive::View is established at the container level, deserializers downstream can and should proceed without checks.
#p The main exception is the higher-level SerializableFormat::Deserialize path, which does perform stream validation in order to support the Format::Decode-style APIs as those are intended to unpack external input data.  See Data::Format and Data::SerializableFormat for more info.

#h1 Ordering and determinism

#p Serialization is order-dependent.
#p The reader must deserialize values in the exact order and type they were written.

#p This explicitness is intentional:
<list>
#p No reflection or schema lookup.
#p No runtime type ambiguity.
#p No hidden allocations. 
</list>

#p This makes the system well suited for file formats, IPC, network packets, caches, and other performance-sensitive data paths.


#h1 Strings and encoding

#p String serialization is explicit.
#p Use SerializeUTF8 / DeserializeUTF8 to encode wide strings into UTF-8 byte sequences.

#p Encoding is handled deterministically and is independent of platform wchar size.


#h1 Relationship to PropertySet

#p Use direct serialization when the data layout is known and fixed.
#p Use PropertySet serialization (EncodePropertySet) when flexibility, flexible version evolution, or dynamic composition is required.

#p Both systems share the same Data namespace but target different problem spaces.

#h1 Relationship to Pack / Unpack

#p Data::Pack and Data::Unpack provide a lower-level mechanism for converting a single value into a binary representation and restoring it again.

#p These functions operate on types that have a well-defined, “toll-free” binary representation - meaning the value can be viewed as a contiguous block of bytes without transformation. This typically includes: 
<list>
#p Integral and floating-point types.
#p POD and trivially copyable types.
#p Arrays of such types. 
</list>

<code>
#p UInt32 t;
#p Data::Archive::View view = Data::Pack(t);
#p 
#p auto restored = Data::Unpack<UInt32>(view);
</code>

#p For types with a compile-time known binary size (for example, UInt32 is always 4 bytes), Pack / Unpack and Serialize / Deserialize produce the same binary representation. Internally, Serialize uses Pack / Unpack for these cases.

#p The distinction becomes important for variable-sized data.

#p Serialize is designed for writing *sequences* of values to a stream. As such, it includes any additional information required to reconstruct the data when reading, such as encoding the length of strings or arrays before their contents.

#p In contrast, Pack produces only the binary representation of the value itself. For example, when packing a string, the length is not prepended, as the size of the binary view implicitly defines the data extent.


#h1 Summary

<list>
#p Data::Archive is a simple byte stream for serialization.
#p Serialize / Deserialize write and read strongly-typed values in order.
#p The API is explicit, deterministic, and allocation-aware.
#p String encoding is handled explicitly via UTF-8 helpers.
#p PropertySet serialization serves higher-level, structured data needs.
</list>
";

	Path: ["Reflex", "Data", "Serialization"];
};

<code>

#p CString owned = b;
</code>

#p Explicit copying keeps ownership semantics visible and avoids hidden allocations in performance-sensitive code paths.

The API is designed to simplify correct async usage while keeping control over execution, lifetime, and thread safety explicit.

#p It covers the most common async use cases, including background execution with progress reporting and cancellation, HTTP requests, safe transfer of payloads across thread boundaries, and lifetime management of running tasks.

#h1 Lifetime

#p Async tasks are reference-counted. Clients typically hold a Reference <Async::Task>.  Destroying this reference automatically requests cancellation of the task (assuming one-owner).

#p For custom workers, task cancellation is cooperative. Worker implementations must explicitly observe and respect the task's run flag via calling Cancelled() and if true, return early. In this case typically you should also ctx.SetResult(false) to indicate failure.

#p GLX layout is handled by a layout model attached to each
#p $spc
#link symbol: [$qReflex$q, $qGLX$q, $qObject$q]; text: $qGLX::Object$q;
#p .
</span>
#p By default every Object uses the standard layout model (kStandardLayout), which supports common 'box layout' behaviors: inline flow, inline flex, float (pin), stretch (fill), plus absolute positioning.

<list>
#p Set the parent's flow (how it arranges inline children) via SetFlow.
#p Set a positioning mode for each child (how the parent treats that child) via the AddXXX or EnableXXX helpers.
#p Optionally enable auto-fit on the parent (size-to-content). 
</list>

#p A useful mental model: 
<list>
#p Parents control *flow*.
#p Children control *positioning*. 
</list>


#h1 Layout Models

#p Each Object owns a layout model (GLX::Detail::LayoutModel), which can be changed via Object::SetLayoutModel.
#p The following standard models are provided:

<list>
#p kStandardLayout: single-line inline flow, plus float/overlay, and absolute positioning.
#p kStandardLayoutWrapped: identical behavior, but inline children wrap onto new rows/columns when space runs out.
</list>

#p Custom layout models can be implemented by deriving from GLX::Detail::LayoutModel, but this is a secondary API. Most UIs should use the standard model and helpers described below.


#h1 Parent Flow

#p The parent controls how *inline* children are arranged. This includes the primary axis, optional inversion, and optional centering of the inline group.

<list>
#p GLX::SetFlow(parent, GLX::kFlowX): inline children flow left -> right.
#p GLX::SetFlow(parent, GLX::kFlowY): inline children flow top -> bottom.
#p GLX::kFlowInvert: reverse the flow direction.
#p GLX::kFlowCenter: center the inline group along the primary axis.
</list>

<code>
#p auto panel = New<GLX::Object>();
#p GLX::SetFlow(panel, GLX::kFlowY);                        //vertical stack
#p //GLX::SetFlow(panel, GLX::kFlowY | GLX::kFlowCenter);   //stack centered as a group
</code>

#p Flow only affects children in *inline* mode. Float and absolute children ignore flow entirely.


#h1 Auto-fit (size to content)

#p Auto-fit is enabled by default. This means an object will grow its content size to accommodate its children.
#p A child's contribution to content size is derived from: 
<list>
#p Its style size / max properties.
#p Its render layers (e.g. Text layers naturally size to text).
#p Its own children, if auto-fit is enabled on that child. 
</list>

<code>
#p GLX::EnableAutoFit(panel, true, true);		//fit width + height to children
#p //GLX::EnableAutoFit(panel, true, false);	//fit width only
</code>

#p Auto-fit is commonly disabled on large scrolling containers, but useful for popups, pills, menus, and small panels.


#h1 Child Positioning Modes

#p Each child has a positioning mode that is interpreted by its parent's layout model.
#p In the standard model, this is set either: 
<list>
#p When adding the child via AddInline / AddFloat / AddAbsolute
#p Explicitly via EnableInline / EnableFloat / EnableAbsolute. 
</list>

#p The AddXXX helpers are convenience functions that: 
<list>
#p Configure the child's positioning mode (via the EnableXXX helpers)
#p Attach the child to the parent (via Object::SetParent). 
</list>

#p As positioning mode and position are child-properties, they persist when changing the parent:

<code>
#p auto parent = New<GLX::Object>();
#p auto another_parent = New<GLX::Object>();
#p auto child = New<GLX::Object>();
#p 
#p GLX::AddFloat(parent, child, GLX::kAlignmentTopRight);
#p 
#p child->Detach();
#p child->SetParent(another_parent); //child is still pinned top-right on the new parent
#p 
#p GLX::EnableFloat(child, GLX::kAlignmentBottomLeft);	//now moved to bottom left
</code>


#h2 Inline Positioning

#p Inline children participate in the parent's flow. They are laid out sequentially along the flow axis.
#p The Orientation parameter controls placement on the *orthogonal axis* (perpendicular to flow):

<list>
#p kOrientationNear: near edge (top or left)
#p kOrientationCenter: centered on the cross axis
#p kOrientationFar: far edge (bottom or right)
#p kOrientationFit: stretch to available cross-axis space (default)
</list>

<code>
#p auto row = New<GLX::Object>();
#p GLX::SetFlow(row, GLX::kFlowX);
#p 
#p auto icon = New<GLX::Object>();
#p auto label = New<GLX::Label>(L$qSettings$q);
#p 
#p GLX::AddInline(row, icon, GLX::kOrientationCenter);  //y centered (ortho axis) on row
#p GLX::AddInline(row, label);                          //fit to row height
</code>


#h2 Inline Flex

#p Using flex allows the child to expand along the *primary axis* to consume remaining space.
#p Conceptually similar to “flex: 1”.

<code>
#p auto header = New<GLX::Object>();
#p GLX::SetFlow(header, GLX::kFlowX);
#p 
#p auto title = New<GLX::Label>(L$qProject$q);
#p auto spacer = New<GLX::Object>();
#p auto close = New<GLX::Button>(L$qX$q);
#p 
#p GLX::AddInline(header, title, GLX::kOrientationCenter);
#p GLX::AddInlineFlex(header, spacer);
#p GLX::AddInline(header, close, GLX::kOrientationCenter);
</code>

#p The remaining space is distributed across all children with flex enabled evenly.

#p To achieve an exact 50% / 50% horizontal split, enable flex on both children and disable horizontal auto-fit so their size is not influenced by content:

<code>
#p auto container = New<GLX::Object>();
#p GLX::SetFlow(header, GLX::kFlowX);
#p 
#p auto left = New<GLX::Label>(L$qLeft$q);
#p auto right = New<GLX::Button>(L$qRight$q);
#p 
#p GLX::EnableAutoFit(left, false, true);    //Important: prevent content from affecting width
#p GLX::EnableAutoFit(right, false, true);
#p 
#p GLX::AddInlineFlex(container, left);
#p GLX::AddInlineFlex(container, right);
</code>


#h2 Float Positioning

#p Float positions a child relative to the parent's bounds but removes it from inline flow.  Floated children do not affect layout of inline siblings.

#p Float positioning can be specified using: 
<list>
#p Alignment enum (single value)
#p Two orientations (x, y)
</list>

<code>
#p auto view = New<GLX::Object>();
#p 
#p auto badge = New<GLX::Label>(L$qNEW$q);
#p GLX::AddFloat(view, badge, GLX::kAlignmentTopRight);
#p 
#p auto spinner = New<GLX::Object>();
#p GLX::AddFloat(view, spinner, GLX::kAlignmentCenter);
#p 
#p auto sticky_footer = New<GLX::Object>();
#p GLX::AddFloat(view, sticky_footer, GLX::kOrientationFit, GLX::kOrientationFar);	//note use of 2nd variant with Orientation enum
</code>

#p Orientation values in float mode mean:
<list>
#p Near: anchor to start edge
#p Center: center on that axis
#p Far: anchor to end edge
#p Fit: stretch along that axis 
</list>


#h2 Stretch (fill parent)

#p Stretch is a convenience mode for the most common float case: filling the parent on both axes.
#p It is shorthand for enabling float with kOrientationFit on X and Y.

<code>
#p GLX::AddStretch(parent, child);	//same as GLX::AddFloat(parent, child, GLX::kOrientationFit, GLX::kOrientationFit)
</code>


#h2 Absolute (explicit coordinates)

#p Absolute places a child at an explicit position in the parent's coordinate space.

<code>
#p auto canvas = New<GLX::Object>();
#p auto node = New<GLX::Object>();
#p GLX::AddAbsolute(canvas, node, {120.0f, 80.0f});
</code>

<warn>
#p Genuine use-cases for absolute positioning are very rare, and should generally be avoided.
</warn>

#p The majority of cases where you may think you need absolute positioning are likely covered by use of GLX::ScrollArea / GLX::ZoomArea or custom drawing.
#p Furthermore, typical genuine cases for absolute positioning will require a custom layout model, to ensure the child positions are updated in response to the parent size (during the Align phase)

#h2 kStandardLayoutWrapped

#p To allow inline children to wrap, set the parent's layout model to kStandardLayoutWrapped.
#p Child APIs remain unchanged.

#p Typical use cases: 
<list>
#p Tag strips / pill lists
#p Wrapping toolbars

#p Stylesheets avoid the unpredictable inheritance rules of CSS while enabling reuse and composition via includes, states, inheritance and aliases.

#h1 Stylesheets

#p Load a stylesheet from disk or resources using RetrieveStyleSheet.

#p Whenever a style is set or hot-reloaded, the framework calls OnSetStyle(const Style & style) on that object.

#p Styles support a core set of properties that define sizing, spacing, colors, layers, and state transitions.

#h2 Margin properties

#p The 'margin' and 'padding' properties accept 1, 2, or 4 float values.

#p By default, values use Reflex ordering:
<code>
#p margin: 24;           // all edges
#p margin: 16,8;         // width, height

</code>

<code>

#p The 'color' (or 'colour') property sets the $qpen$q colour, whereas 'bg_color' (or 'bg_colour') defines a rectangular fill colour.  Colors always use 0-255 ranges for components, including the alpha channel.

<code>  
#p colour: 128;                // greyscale

<code>  
#p colour: $FF0000;

#p bg: Border(width: 2; corner: 4; colour: 255,0,0);
</code>

#p The 'opacity' property is a normalized single float value (0.0 - 1.0), applied multiplicatively to all layers and children of an object.

<code>
#p opacity: 0.75;
</code>

#p Each object supports two layer arrays: 'bg' (drawn before children) and 'fg' (drawn after children). Typically use bg, but for cases where rendering $qon top of$q content is required, use fg (e.g. InnerShadow).

#h1 Inheritance and Aliasing

#p Stylesheets support reuse and composition through the inherit property and the special @Alias directive.

#p Using inherit, a style copies all properties from another style before applying its own overrides. This allows you to define a base style and then extend or specialize it with minimal duplication:

<code>
#p BigButton:
#p { 
#p 	inherit: Button;
#p 	size: 128;
#p }; 
</code>

#p The inherit property accepts either a single style ID, or a path to a nested sub-style. For example, to inherit from a sub-style defined inside another style:

<code> 
#p inherit: Dialog > Header;
</code>

#p Additionally, the @Alias style variant lets you reuse a style as a named property of another style. Unlike inherit, aliasing does not copy values, and instead directly references the source definition:

<code> 
#p Popup:
#p {
#p 	inherit: Button;
#p 	@Alias menu; // alias a menu property defined previously

#p 	}
#p
#p 	return Base::OnEvent(src, e);
#p }
</code>

#p If you want to prevent this object and its parents from accepting the current drag, you can intercept the tender event, clear its id, and then allow it to continue upward as a non-drag event:

<code>
#p case GLX::kDragDropTender:
#p 	e.id = kNullKey;	//change the Event id to effectively hide the event from parents
#p 	return false;	//return false as this object doesnt want to receive the drop
</code>

#h1 Reading drag data

#p QueryDragDropData<TYPE>(e) is the standard typed helper. Internally it performs a DynamicCast on the object stored in the event's drag payload.

<code>
#p struct CustomDragData : public Reflex::Object
#p {
#p 	REFLEX_OBJECT(CustomDragData, Reflex::Object);
#p
#p 	Array <WString> filepaths;
#p };
#p
#p template <class TYPE> inline TYPE * Reflex::GLX::QueryDragDropData(GLX::Event & e)
#p {
#p 	return DynamicCast<TYPE>(GetDragDropData(e));
#p }
</code>

#p If you use a custom payload type, make it object-castable with REFLEX_OBJECT so DynamicCast can recognise it.

#p When checking against multiple possible payload types, it is slightly more efficient to fetch the payload once and then test it manually:

<code>
#p auto drag_data = GLX::GetDragDropData(e);
#p
#p if (auto custom = DynamicCast<CustomDragData>(drag_data))
#p {
#p }
#p else if (auto generic = DynamicCast<Data::PropertySet>(drag_data))
#p {
#p }
</code>

#h1 Cursor feedback

#p The drag API takes two cursors: the cursor shown while hovering an accepting target, and the cursor shown when the current target does not accept the drag. A simple setup is:

<code>
#p GLX::StartDragDrop(data, GLX::kMouseCursorPointer, GLX::kMouseCursorBlock);
</code>

#p For richer UI, a common pattern is to hide the OS drag cursor and render your own drag-preview object in the window foreground. Start the drag with invisible cursors:

<code>
#p GLX::StartDragDrop(data, GLX::kMouseCursorInvisible, GLX::kMouseCursorInvisible);
</code>

#p A typical custom-cursor implementation uses a global begin-listener to construct the preview from the drag payload type, attaches that object to the window foreground, disables mouse handling on it, and updates its position every frame.
#p A key rule is to exclude the visual cursor from hit-testing. Use GLX::EnableMouse(cursor, false, true) so it ignores mouse-over events.

<code>
#p m_drag_drop_visualiser = GLX::CreateDragDropBeginListener([this](Reflex::Object & drag_data)
#p {
#p 	auto origin = GLX::GetFocus();
#p 	TRef <GLX::WindowClient> window = origin->GetWindow();
#p 
#p 	//create drag cursor
#p 	auto cursor = New<GLX::Object>();
#p 	GLX::SetText(cursor, GLX::GetText(origin));	
#p 	cursor->SetStyle(GLX::FindStyle(origin, $qDragCursor$q));
#p 	GLX::EnableMouse(cursor, false, true);	//ensure cursor, or any children on cursor, do not doesnt intercept mouse (see EnableMouse for details)
#p 	GLX::Enter(cursor, GLX::kEnterAnimationFade);
#p 	GLX::AddAbsolute(window->GetForeground(), cursor, GLX::GetPointerPosition(window));
#p 
#p 	//attach callbacks to window, so if window is destroyed they wont be called
#p 	SetAbstractProperty(window, $qdragdrop_clock$q, GLX::CreateAnimationClock([window, cursor](Float)
#p 	{
#p 		cursor->SetPosition(GLX::GetPointerPosition(window));
#p 	}));
#p 	auto run_opacity_animation = [cursor](GLX::Object & drop_target)
#p 	{
#p 		auto fade = GLX::CreateOpacityAnimation($qopacity$q, 0.75f, 0.9f);
#p 		if (IsNull(drop_target)) fade->Flip();
#p 		GLX::Run(cursor, $qopacity$q, 0.25f, fade);
#p 	};
#p 	run_opacity_animation(Null<GLX::Object>());	//call now to apply initial fade to cursor
#p
#p 	SetAbstractProperty(window, $qdragdrop_target$q, GLX::CreateDragDropTargetListener([this, run_opacity_animation](GLX::Object & drop_target)
#p 	{
#p 		m_drop_target.Load()->UnsetState($qdragover$q);	//using Reflex::Detail::WeakRef to cover case drop_target might be stack-based
#p 		m_drop_target.Store(drop_target);
#p 		drop_target.SetState($qdragover$q);
#p 
#p 		run_opacity_animation(drop_target);
#p 	}));
#p 	SetAbstractProperty(window, $qdragdrop_end$q, GLX::CreateDragDropEndListener([this, window, cursor]()
#p 	{
#p 		GLX::Exit(cursor, true);	//detach cursor with fade
#p 		
#p 		m_drop_target.Clear();
#p 		
#p 		UnsetAbstractProperty(window, $qdragdrop_clock$q);
#p 		UnsetAbstractProperty(window, $qdragdrop_target$q);
#p 		UnsetAbstractProperty(window, $qdragdrop_end$q);	//important always delete containing lambda last
#p 	}));
#p });
</code>

#p To support multiple payload types, a convenient pattern is to register a small factory per drag-data type and choose the preview object in CreateDragDropBeginListener. This lets file drags, object drags, and generic property payloads each supply their own visual cursor.

<warn>
#p In a multi-instance plugin environment, take care to do this once, otherwise each window would create a mousecursor.
</warn>

	@WString Description: "#p Reflex makes a primary, explicit distinction between object-types and value-types. All object-types derive from Reflex::Object, are shared, reference counted, typically heap-based (although they can also be instantiated on the stack), and non-copyable by default.

#p Value types (non-Objects such as UInt32 or Array) are lightweight stack-based primitives. To promote a value into an object so it can participate in reference counting, heap allocation, dynamic properties, or generic object APIs, use ObjectOf<TYPE>.

#p Reflex::Object provides two foundational capabilities:

<list>
	#p Deterministic lifetime management via intrusive reference counting
	#p A generic interface for attaching and querying typed properties dynamically
</list>

#h1 Object Lifetime

#p Reflex uses an intrusive reference counting model built directly into Object.

#p Unlike std::shared_ptr, COM, or Objective-C, there are no external control blocks or hidden allocations - the reference count lives inside the object itself.

#h2 Core Principles

<list>
	#p All reference-counted types derive from Object
	#p The reference count is stored inside the object itself
	#p Newly created objects always start with a reference count of zero
	#p Every Release() must therefore correspond to a previous Retain()
	#p Ownership semantics are primarily expressed through Reference<T> and TRef<T>
</list>

#h2 Reference <TYPE>

#p Reference<T> is the primary ownership wrapper for Reflex objects. It retains on construction, releases on destruction, and guarantees a valid object reference over its lifetime.

<span>
#p See$spc
#link symbol: [$qReflex$q, $qReference$q]; text: $qReflex::Reference$q;
#p $spcfor full semantics and usage.
</span>

#h2 TRef <TYPE>

#p TRef<T> is the lightweight non-owning counterpart to Reference<T>. It does not retain or release, but still follows the Reflex convention of representing a valid object reference rather than a nullable pointer.

<span>
#p See$spc
#link symbol: [$qReflex$q, $qTRef$q]; text: $qReflex::TRef$q;
#p $spcfor full semantics and usage.
</span>

#h2 Object Creation

#p Reflex objects should never be created with raw new/delete directly.

#p Instead, object creation is performed through the Reflex helper APIs:

<code>
#p auto a = New<MyClass>(args...);
#p auto b = Make<MyClass>(args...);
</code>

<list>
	#p New<T>() returns a TRef<T>
	#p Make<T>() returns a Reference<T>
	#p If T is abstract, New<T>() automatically resolves through T::Create()
</list>

#p Make<T>() is generally the most convenient form when immediate ownership is required.

#h2 AutoRelease Helpers

#p AutoRelease() provides a shorthand for constructing temporary Reference<T> wrappers.

#p The following examples are equivalent:

<code>
#p Reference<HttpConnection> conn = HttpConnection::Create(...);
#p auto conn = AutoRelease(HttpConnection::Create(...));
#p auto conn = Make<HttpConnection>(...);
</code>

#h2 Stack-Allocated Objects

#p Unlike traditional COM-style systems, Reflex objects may be instantiated directly on the stack.

<code>
#p Data::PropertySet node;
</code>

#p Stack objects ignore Retain() and Release() calls and are destroyed normally via scope lifetime.  

#p This flexibility is useful and efficient for temporaries and embedded members, however ownership must remain explicit: any retained reference must not outlive the stack object it refers to. For example, storing a Reference or TRef to a stack object beyond its lifetime is invalid:

<code>
#p Data::PropertySet node;
#p auto props = New<Data::PropertySet>();
#p props->SetProperty($qchild$q, node); // invalid after node leaves scope
</code>

#h2 Null Instances

#p Reference<T> and TRef<T> default construct using the type's null-instance rather than nullptr.

#p If a type does not define a null-instance, the wrappers cannot be default-constructed, enforcing validity at compile time.

#p User-defined object types therefore often require explicit construction:

<code>
#p Reference <View> view1 = New<View>();
#p Reference <View> view2 = kNewObject;	//shorthand helper
</code>

#p TRef<T> also supports construction using kNoValue for deferred assignment, however this should only be used sparingly.

#h2 Manual Retain/Release

#p Object exposes Retain() and Release(), however direct usage is uncommon.

#p If manual lifetime management is required:

<list>
	#p New objects begin with refcount = 0
	#p Every Release() must correspond to a previous Retain()
	#p Double-release will assert
	#p Retaining in constructors and releasing in destructors is the standard pattern
</list>

#h2 Circular References

#p Reflex is not a garbage collected system. Circular references will leak unless explicitly broken:

<code>
#p auto a = New<Data::PropertySet>();
#p auto b = New<Data::PropertySet>();
#p a->SetProperty($qfoo$q, b);
#p b->SetProperty($qfoo$q, a);
</code>

#p Some advanced systems inside Reflex (such as Reflex::VM) provide explicit cycle-breaking mechanisms, however application code should generally avoid circular ownership through design.

#h2 Best Practices

<list>
	#p Prefer Reference<T> for ownership
	#p Use TRef<T> for temporary non-owning references
	#p Avoid manual Retain()/Release() where possible
	#p Avoid circular ownership relationships
	#p Prefer clear ownership hierarchies
</list>

#p Signal and listener lifetimes are fully independent: it is safe for a Signal to be destroyed before its listeners, and for listeners to be destroyed before the Signal.

#h2 Dispatch Safety

#p At the start of a notification, the current listener list is copied into a temporary dispatch buffer using strong references. This guarantees safe iteration in the presence of re-entrancy and list mutation.

<list> 
#p Listener attachment and removal during Notify(...) is always safe. 
#p Re-entrant calls to Notify(...) are supported. 
#p The dispatch set is stable for the duration of the call. 
</list>

#p Listeners added while a Notify(...) call is in progress will not receive that notification, and will be invoked starting from the next Notify(...).

#p Listener removal during dispatch takes effect immediately: if a listener is detached or released such that it is only retained by the internal dispatch buffer, it is skipped and will not be invoked.

#h2 Signal::Mute
#p Supports temporary muting via Signal::Mute.

#p If a signal is muted, notifications are suppressed until the mute scope is released. Nested mutes are supported.

	Group: true;

	@WString Description: "#p Reflex GLX provides three canvas entry points for custom, code-driven drawing, ordered from simplest and fastest to most advanced:

<list>
#p SetCanvas for monochrome geometry, using Array<Point> output.
#p SetColourCanvas for multi-colour geometry, using Array<ColourPoint> output.
#p SetGraphicCanvas for advanced cases where you need direct access to graphics, images, fonts, or a custom Graphic that can change every frame.
</list>

#p In stylesheets, these correspond to:

<list>
#p bg: Canvas();
#p bg: ColourCanvas();
#p bg: GraphicCanvas();
</list>

#h1 Execution model

#p These canvas callbacks are not immediate-mode draw calls in the traditional sense. They are not invoked every frame just because the object is visible.

#p Instead, the callback prepares geometry for the renderer, effectively building VBO data that can be reused until something changes.

#p To request a new canvas build, call Redraw() on the object. For example:

<code>
#p m_object.Redraw();
</code>

#p Other state changes that already imply a visual refresh, such as Update(), Realign(), and related layout-reaccommodation paths, also schedule a redraw.

#p For animated content, a typical pattern is to trigger redraws from OnClock(Float):

<code>
#p void MyView::OnClock(Float dt)
#p {
#p 	Redraw();
#p }
</code>

#h1 Choosing the right API

#p Prefer Canvas over ColourCanvas over GraphicCanvas, in that order, for performance and simplicity.

#p Use Canvas() when you are drawing simple monochrome paths and shapes.
#p Use ColourCanvas() when your geometry needs per-point colour.
#p Use GraphicCanvas() only when you need the extra power geometry plus textures (eg image/font) composition (which are non primary APIs).

#h1 Style usage

#p Canvas bindings are usually declared in styles with an optional id. Use the plain form when there is only one binding for the object:

<code>
#p bg: Canvas();
</code>

#p When you have multiple canvas bindings on the same object, give each one an id and bind them explicitly in code:

<code>
#p bg: Canvas(id: logo), Canvas(id: clouds);
#p
#p SetCanvas(m_obj, {.id = K32($qlogo$q)}, draw_logo);
#p SetCanvas(m_obj, {.id = K32($qclouds$q)}, draw_clouds);
</code>

#p The same pattern applies to colour and graphic canvases:

<code>
#p SetColourCanvas(m_obj, {}, draw_coloured_geometry);
#p SetGraphicCanvas(m_obj, {}, draw_advanced_graphic);
</code>

#p For Canvas and ColourCanvas, do not clear ctx.output. Append to it only. The system may batch multiple canvases together and compact them into a single VBO.

#h1 Simple Demo

#p A minimal Canvas demo can draw a star shape by appending a closed path to ctx.output:

<code>
#p void StarView::OnSetStyle(const GLX::Style & style)
#p {
#p 	GLX::SetCanvas(*this, {}, [](GLX::CanvasContext & ctx)
#p 	{
#p 		auto cx = ctx.size.w * 0.5f;
#p 		auto cy = ctx.size.h * 0.5f;
#p 		auto outer = Min(ctx.size.w, ctx.size.h) * 0.42f;
#p 		auto inner = outer * 0.45f;
#p 		GLX::Point star[10];
#p
#p 		for (UInt i = 0; i < 10; i++)
#p 		{
#p 			auto a = -kPif * 0.5f + kPif * 0.2f * Float(i);
#p 			auto r = (i & 1) ? inner : outer;
#p 			star[i] = { cx + Cos(a) * r, cy + Sin(a) * r };
#p 		}
#p
#p 		GLX::AddPath(ctx.output, star, true, 2.0f, GLX::kPathJoinRound, GLX::kPathCapRound);
#p 	});
#p }
</code>

#p This demo uses SetCanvas because the geometry is monochrome. As with most layers, it will be drawn at the style colour, modulated by its own colour property.  Use ColourCanvas only when you need per-point colour, and keep GraphicCanvas for the advanced cases where you need the full Graphic pipeline.

#h1 Example project

#p See examples/Custom Drawing for a practical demonstration of the Canvas API in use.
";

#p 		//update cached output gain

#p The 'lock' argument should typically be left true.

#p When enabled, Reflex requests platform locking or restricted share access so concurrent access is reduced while the handle is open.

<list>

#p For kModeOverwrite and kModeAppend this is typically an exclusive write lock.
#p With 'lock' set false, the remaining sharing behaviour is platform-specific and other processes may still read or write the file concurrently.
</list>

#p This setting controls concurrent access, not write durability or synchronisation to disk.
